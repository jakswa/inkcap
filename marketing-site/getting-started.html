Getting started

inkcap is one stateful process: web server plus agent runtime. Bring PostgreSQL and an LLM provider; the server keeps runs moving even when every browser disconnects.

Fast path: Docker + existing Postgres

Create an inkcap database, then run the published GHCR image. This example points at PostgreSQL on the host machine and also allows local providers such as llama-server at host.docker.internal.

docker run -d --name inkcap --restart unless-stopped \
  -p 3000:3000 \
  --add-host=host.docker.internal:host-gateway \
  -e DATABASE_URL=postgresql://postgres:postgres@host.docker.internal:5432/inkcap \
  -e SESSION_SECRET="$(openssl rand -base64 32)" \
  -e REGISTRATION=open \
  -e OUTBOUND_TRUSTED_HOSTS=host.docker.internal \
  ghcr.io/jakswa/inkcap:latest \
  sh -lc 'bun build/tasks/migrate.js && exec bun build/index.js'

Open http://localhost:3000, register, add a provider, and start chatting. For a real deployment, use a fixed secret, keep registration closed, and bootstrap the first user with the task below.

Source checkout

Use this path for hacking on inkcap or running tests.

git clone https://github.com/jakswa/inkcap
cd inkcap
bun install
cp .env.example .env.local   # set DATABASE_URL and SESSION_SECRET
bun run db:migrate
bun run dev

Add a provider

Providers live server-side; API keys are stored in PostgreSQL and never reach the browser. Add one under Providers → Add provider. Supported paths include llama-server, OpenAI-compatible APIs, and ChatGPT via Codex.

# source checkout helper; uses DEV_LLAMA_SERVER / DEV_LLAMA_KEY
bun src/tasks/seed-provider.ts --user you@example.com

Chat on your ChatGPT subscription (Codex)

Providers → Add provider → “Sign in with ChatGPT”. inkcap uses Codex device-code auth by default: approve the one-time code in any browser, then tokens stay server-side and refresh automatically.

You may need to enable Allow device code login in ChatGPT settings or workspace policy. The old localhost:1455 PKCE callback remains as an advanced fallback, including paste-to-finish for remote browsers.

Import llama.cpp web UI history

The importer is idempotent and accepts JSONL or zip exports, including attachments and branch trees:

bun src/tasks/import-llama-ui.ts export.jsonl --user you@example.com

Environment reference

VariableWhat it does
DATABASE_URLPostgreSQL connection string
SESSION_SECRETEncrypts the session cookie; production needs a fixed, non-placeholder value at least 32 bytes long
ASSET_VERSIONStatic-asset version; required when building a production image
PORTHTTP port inside the container/process (default 3000)
REGISTRATIONopen or closed; defaults to closed in production, open elsewhere
PUBLIC_ORIGIN, CSRF_TRUSTED_ORIGINS, OUTBOUND_TRUSTED_HOSTSSplit-origin and local-network deployment opt-ins; see .env.example
DEV_LLAMA_SERVER / DEV_LLAMA_KEYFeed the development seed-provider task

Production notes

Every push to master builds a multi-arch image at ghcr.io/jakswa/inkcap:latest plus a SHA tag. To self-build instead:

docker build --build-arg ASSET_VERSION=$(git rev-parse --short=7 HEAD) -t inkcap .

The quick-start command migrates on container start. If you run the default image command instead, run migrations with the same environment as the app:

docker exec inkcap bun build/tasks/migrate.js

Registration defaults to closed in production. Bootstrap the first user from the running container:

docker exec -e CREATE_USER_PASSWORD=... inkcap \
  bun build/tasks/create-user.js --name You --email you@example.com
inkcap is deliberately a single stateful process. Restarts are safe: boot recovery finalizes runs interrupted mid-generation.
