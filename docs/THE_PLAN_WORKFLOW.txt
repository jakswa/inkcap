The workflow proposal

Wave 1 — Foundations (mostly parallel, cheap):
- schema (opus, worktree): ALL v1 migrations in one hand — providers, conversations, messages (tree via parent_id), runs, attachments — plus typed queries and query tests. One agent owns migration numbering 002–005 so nobody ever merge-conflicts on SQL files. This is the load-bearing agent of the whole build.
- organ-harvest (sonnet, read-only, no worktree): strip-mines ~/sandbox/llama-ui into spec documents: exact completion-request shape, llama-server extras (/props, error-body context parsing), the JSONL/zip export format with real sample payloads. Pure research; feeds waves 2–3 prompts.
- m0-flake (haiku/sonnet): root-cause the concurrent-test collision, fix, so every later agent trusts red tests.

Wave 2 — The boring parallel middle (sonnet, one worktree + own test DB each):
- providers: SSR CRUD + connection test + seed-from-DEV_LLAMA_* task → M1.
- chat-skeleton: conversation list/show/new + form-POST send with a naive blocking completion call → M2.
- markdown: server-side renderMarkdown() — parse, highlight, sanitize — as a pure, heavily-tested util (M4's core, zero dependencies on the others).
- importer: JSONL/zip parser + import task, built against organ-harvest's samples → M5's guts.

These four touch disjoint files (schema already landed), so worktrees are genuinely conflict-free, not just isolated.

Wave 3 — The runner (serial, opus, high effort): streaming, debounced persistence, SSE replay-from-cursor, boot recovery, stopping points. One agent (or me inline), then a parallel adversarial verify panel: independent agents that literally kill -9 the server mid-stream, reconnect SSE with stale cursors, race two sends on one conversation, and try to refute the recovery claims. This is where multi-agent buys correctness, not just speed.

Wave 4 — Rich view + integration polish (parallel again): the vanilla-JS chat island, finalize-swap rendering, mobile layout; plus a completeness critic against THE_PLAN.md.

Mechanics

- Test DBs: the starter requires the test DB name to end in test, so each worktree agent gets spail_<agent>_test via an uncommitted .env.test tweak, created/dropped by the agent. No shared state.
- queries.gen.d.ts is never hand-merged — I regenerate at each integration point.
- I stay the integrator: agents finish work in their worktrees; between waves I merge, regen types, run the full suite, and make the milestone commits — so your one-commit-per-milestone history survives a parallel build.
- Model split, as you suggested: sonnet for CRUD/pure utils, opus for schema and the runner, verify panel mixed.

Wave 1 is cheap and de-risks everything downstream. Say go and I'll launch it — or "go all the way" and I'll chain waves autonomously, checkpointing at each milestone commit.
